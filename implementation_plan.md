# 竞彩 +EV 分析工具实现计划

## 1. 背景与目标

本计划用于把现有的竞彩“赔率冰冻期”正期望值（+EV）策略，整理成一个可落地的本地分析工具方案。

核心思路是：在中国竞彩赔率进入临场冻结或低频更新阶段后，对比国际市场（如 Pinnacle 或第三方赔率源）的实时赔率变化，通过去抽水后的公允概率计算竞彩侧是否存在正期望值机会。

第一版目标不是自动下注，也不是做高频爬虫系统，而是建立一个可复用、可审计、可由 Codex、Trae、Hermes 等 agent 调用的本地分析流程：

- 使用真实浏览器低频获取赔率数据，降低被网站反爬误伤的风险。
- 将每次获取的数据保存为 JSON 快照，保留时间戳和来源。
- 用确定性代码完成赔率标准化、EV 计算和组合筛选。
- 用 skill 或类似提示层让 agent 负责流程编排、结果解释和风险提醒。

## 2. 推荐架构

推荐采用 **CLI 核心 + Skill 薄封装** 的架构。

### 2.1 为什么不让 agent 直接计算

不建议把核心计算完全交给 agent 临场完成，原因包括：

- 赔率、概率、EV、凯利比例属于高精度数值计算，必须稳定、可复现。
- 赛事匹配、玩法映射、组合过滤容易出现边界错误，需要测试覆盖。
- 每次分析都应能追溯输入 JSON、计算参数和输出结果。
- 不同 agent 的上下文、模型版本和推理表现不同，不能依赖自然语言推理保证一致性。

因此，核心计算应写成确定性代码，由 CLI 执行；agent 只负责调用、解释、复核和提醒。

### 2.2 推荐模块划分

建议将系统拆成四层：

1. **数据获取层**
   - 使用真实浏览器打开目标页面。
   - 低频读取页面上的赔率数据。
   - 保存原始抓取结果和必要的调试信息。

2. **数据标准化层**
   - 将竞彩、Pinnacle 或其他数据源转换成统一结构。
   - 统一赛事 ID、主客队、开赛时间、玩法、选项和赔率字段。
   - 标记无法匹配或需要人工复核的数据。

3. **计算分析层**
   - 对国际赔率去抽水，得到公允概率。
   - 计算竞彩各选项单项 EV。
   - 过滤正 EV 单项。
   - 生成跨场 2 串 1 组合。
   - 计算组合 EV 和分数凯利比例。

4. **Agent / Skill 编排层**
   - 告诉 agent 何时运行抓取和分析。
   - 指定 CLI 命令、输入输出路径和结果解释方式。
   - 提醒数据时效、反爬风险、匹配失败风险和不得自动下注。

## 3. 数据获取方案

第一版采用 **真实浏览器优先** 的数据获取方式。

### 3.1 浏览器优先

真实浏览器方案适合初期落地：

- 使用本机浏览器或可见浏览器会话打开目标页面。
- 保留正常用户访问节奏，避免高频请求。
- 每次读取后保存 JSON 快照，而不是只保留最终报告。
- 页面无法访问、验证码、登录限制或结构变化时，停止自动流程并提示人工处理。

默认频次建议：

- 手动双频模式：每天运行 1-2 次。
- 临场监控模式：只在指定时间窗口内低频运行，例如 60-180 秒一次。
- 第一版不建议 30 秒级别持续抓取，除非后续接入稳定 API 或获得明确授权的数据源。

### 3.2 API 备用

后续可接入第三方赔率 API 或官方授权 API：

- API 返回结构更稳定，适合自动化和历史回测。
- 成本可能更高，但被封风险更低。
- 若 API 可用，应优先用于生产化流程，真实浏览器作为补充校验。

### 3.3 手动录入备用

当网页反爬、页面结构变化或 API 不可用时，应允许手动录入 JSON 或表格数据。

手动录入模式适合：

- 验证计算模型。
- 构造测试样本。
- 在自动取数失败时保留人工分析能力。

## 4. 数据文件结构

建议将数据按处理阶段分为三类目录。

```text
data/
  raw/
    2026-06-25_213000_sporttery.json
    2026-06-25_213000_pinnacle.json
  normalized/
    2026-06-25_213000_matches.json
  analysis/
    2026-06-25_213000_ev_report.json
    2026-06-25_213000_ev_report.md
```

### 4.1 raw

`raw/` 保存原始提取结果。

每个文件至少包含：

- `source`：数据来源，例如 `sporttery`、`pinnacle`、`manual`。
- `url`：数据页面或接口地址。
- `fetched_at`：抓取时间。
- `raw_payload`：原始页面解析结果或 API 返回结果。
- `notes`：异常说明，例如页面结构变化、部分字段缺失。

### 4.2 normalized

`normalized/` 保存标准化后的赛事和赔率数据。

核心字段建议包括：

- `match_id`
- `source_match_id`
- `home_team`
- `away_team`
- `start_time`
- `market_type`
- `handicap`
- `outcomes`
- `odds`
- `source`
- `matched_status`

### 4.3 analysis

`analysis/` 保存计算结果。

结果应包含：

- 输入数据文件路径。
- 计算参数，例如 EV 阈值、凯利 fraction。
- 单项 EV 列表。
- 2 串 1 候选组合。
- 无法匹配的数据。
- 风险提示。
- 报告生成时间。

## 5. 核心计算规则

第一版采用简单、透明、可测试的计算方法。

### 5.1 国际赔率去抽水

对同一玩法下的互斥选项赔率 `O_i`，先计算庄家 margin：

```text
margin = sum(1 / O_i) - 1
```

再用比例法计算公允概率：

```text
fair_probability_i = 1 / (O_i * (1 + margin))
```

比例法不是唯一去抽水方法，但第一版足够清晰，便于验证。后续可增加 Shin 方法或指数法作为可选参数。

### 5.2 单项 EV

将国际赔率得到的公允概率与竞彩赔率对比：

```text
single_ev = fair_probability * sporttery_odds - 1
```

默认只保留：

```text
single_ev > 0
```

### 5.3 2 串 1 组合

候选组合必须满足：

```text
match_id_a != match_id_b
```

组合 EV 计算：

```text
combo_ev = (1 + single_ev_a) * (1 + single_ev_b) - 1
```

第一版默认组合阈值：

```text
combo_ev >= 0.08
```

### 5.4 分数凯利

对通过阈值的组合计算建议资金比例：

```text
kelly_fraction = 0.1
stake_ratio = kelly_fraction * combo_ev / (combo_odds - 1)
```

输出只能作为风险参考，不应被解释为自动下注指令。

## 6. Skill 设计

Skill 的职责是“让 agent 正确使用工具”，而不是承载核心计算。

### 6.1 Skill 应包含的内容

Skill 文档应说明：

- 什么时候使用该技能。
- 运行前需要确认哪些数据源。
- 如何调用 CLI 抓取数据。
- 如何调用 CLI 生成分析报告。
- 如何阅读报告中的 EV、组合 EV、凯利比例和风险提示。
- 何时必须要求人工复核，例如赛事匹配失败、页面数据缺失、抓取时间过旧。

### 6.2 Skill 不应承担的内容

Skill 不应：

- 手写核心计算公式并要求 agent 临场计算。
- 自动下注。
- 绕过网站访问限制。
- 诱导高频抓取。
- 在数据不完整时给出确定投注建议。

### 6.3 跨工具兼容

为兼容 Codex、Trae、Hermes，核心能力应通过 CLI 暴露，而不是绑定某一个 agent 平台。

不同平台只需要各自维护一份轻量调用说明：

- Codex：用 skill 调用 CLI。
- Trae：用项目规则或 agent prompt 调用 CLI。
- Hermes：用工具说明或工作流配置调用 CLI。

## 7. 风控与限制

### 7.1 反爬与访问限制

真实浏览器可以降低自动脚本被拦截的概率，但不能保证稳定。

第一版必须遵守以下原则：

- 不破解验证码。
- 不绕过登录、地域或访问限制。
- 不做高频请求。
- 不伪造大规模访问行为。
- 页面不可访问时停止流程，并提示人工处理。

### 7.2 页面结构变化

网页字段、DOM 结构、玩法名称或赔率展示方式可能变化。

处理原则：

- 解析失败时不生成正常报告。
- 保留失败快照或截图，方便后续修复。
- 报告中明确列出缺失字段和无法匹配项。

### 7.3 赛事匹配失败

不同数据源对球队名称、赛事名称和开赛时间的表示可能不同。

必须避免静默错配：

- 匹配置信度不足时进入人工复核。
- 同名球队、青年队、女足、友谊赛等场景需要特别谨慎。
- 不允许把无法确认的匹配用于 EV 计算。

### 7.4 数据时效

赔率数据高度依赖时间。

报告中必须展示：

- 竞彩数据时间。
- 国际赔率数据时间。
- 两者时间差。
- 报告生成时间。

当数据过旧时，报告应标记为不可用于临场决策。

### 7.5 不自动下注

第一版只生成分析报告，不执行下注、不提交订单、不联系彩店。

任何投注决策都必须由人工基于最新数据、实际销售状态和个人风险承受能力自行判断。

## 8. 第一版范围

第一版只做最小可用版本。

### 8.1 支持玩法

优先支持：

- 胜平负。
- 让球胜平负。
- 总进球数。

### 8.2 支持流程

第一版支持：

- 真实浏览器低频获取数据。
- 保存 JSON 快照。
- 标准化赛事与赔率。
- 计算单项 EV。
- 生成 2 串 1 候选组合。
- 输出 JSON 和 Markdown 报告。

### 8.3 暂不支持

第一版不支持：

- 自动下注。
- 高频监控。
- 实时推送。
- 历史回测。
- 多数据源加权模型。
- 复杂资金管理策略。

## 9. 测试与验收

### 9.1 样例 JSON 测试

构造固定样例，验证：

- 去抽水后概率之和约等于 1。
- 单项 EV 正负判断正确。
- 2 串 1 组合 EV 计算正确。
- 同一场比赛的不同选项不会被组合。
- 分数凯利比例计算正确。

### 9.2 真实小样本测试

使用少量真实数据验证：

- 竞彩与国际数据能正确匹配。
- 玩法和选项映射正确。
- 无法匹配的数据会进入复核列表。
- 报告中的时间戳和来源清晰。

### 9.3 浏览器失败场景测试

模拟或记录以下失败场景：

- 页面打不开。
- 页面出现验证码。
- 页面结构变化。
- 部分赔率缺失。
- 数据源时间差过大。

验收标准是：系统不应生成误导性的正常分析报告，而应明确报错并保留可复核信息。

## 10. 后续扩展

在第一版稳定后，可以逐步扩展：

- 接入第三方赔率 API，降低浏览器抓取依赖。
- 增加历史数据存档和回测。
- 增加报告界面或本地仪表盘。
- 增加定时提醒，但仍保持低频和人工确认。
- 增加 Shin 去抽水法、指数法等概率模型。
- 增加多庄家赔率对比，降低单一数据源偏差。
- 增加赛事名称映射表和人工校正机制。

## 11. 默认决策

本计划采用以下默认决策：

- **实现形态**：CLI 核心 + Skill 薄封装。
- **数据获取**：真实浏览器优先，API 和手动录入备用。
- **数据格式**：所有中间结果和最终结果都保存为 JSON。
- **计算方式**：代码确定性计算，不让 agent 临场心算。
- **输出方式**：生成 JSON 和 Markdown 报告。
- **安全边界**：不自动下注，不高频抓取，不绕过访问限制。
